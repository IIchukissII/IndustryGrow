# Carrier PCB — STM32F405 pin map

- **Document type:** Hardware reference — pin map (not an ADR)
- **Date:** 2026-05-25
- **Author:** Romanchuk
- **Project:** IndustryGrow
- **Subject:** Signal pin assignment for the IndustryGrow sensor-node carrier PCB
- **MCU:** STM32F405RGT6 (LQFP64) on a WeAct STM32F4 64-Pin Core Board
- **References:** ADR-0002 rev 3 (field bus architecture), ADR-0014 (sensor node taxonomy / sensor-module header), ADR-0007 (PKI, ATECC608)
- **Sources of truth:** AF mappings — STM32F405xx datasheet DS8626 Rev 9, Table 9. Board pin occupancy — WeAct STM32F4 64-Pin Core Board V1.1 schematic.

## Scope

This document fixes the pin assignment for the sensor-node carrier PCB. The carrier hosts a WeAct STM32F4 64-Pin Core Board and routes the STM32F405 signals to the CAN transceiver, the ATECC608 secure element, the sensor-module headers (ADR-0014 decision 5), and status indication. SWD programming uses the WeAct board's own debug header and is not routed through the carrier (see notes).

Every alternate-function (AF) assignment in the table below has been cross-checked against STM32F405 datasheet Table 9. Pin budget: 27 functional signals routed by the carrier, of ~49 available LQFP64 GPIO; 7 spare. PA13/PA14 (SWD) are listed for reference but are reached on the WeAct debug header, not through the carrier.

## WeAct board — pins not available for carrier signals

These pins are wired to peripherals on the WeAct core board itself. CubeMX validates AF mappings at the chip level only and has no knowledge of the WeAct board, so it will **not** flag these — the constraint is board-level. Do not assign carrier signals to them.

| Pins | Wired on the WeAct board to |
|---|---|
| PA11, PA12 | USB Type-C data lines (OTG_FS_DM / OTG_FS_DP) |
| PC14, PC15 | LSE crystal, 32.768 kHz (RTC) |
| PC8, PC9, PC10, PC11, PC12, PD2 | microSD slot (SDIO) |
| PA8 | microSD slot support |
| PC13 | on-board user button |
| PB2 | on-board blue LED (also BOOT1) |

PA13 / PA14 are the SWD pins. On the WeAct board they are brought out on a separate debug header (alongside NRST, GND, 3V3 and the USART pins), **not** on P1/P2 — so the carrier does not route SWD. Program and debug the assembled node by attaching an ST-Link to that debug header directly; it stays accessible with the carrier mounted. Do not assign other carrier signals to PA13/PA14.

## Power — WeAct board connectors

Power and ground on the WeAct board are **not** carried on the P1/P2 signal headers (apart from GND, and VB on P1.1). They are on two dedicated power headers at the bottom of the board:

| Header | Size | Signals | Carrier handling |
|---|---|---|---|
| P3 | 2×3 (6 pins) | 3V3, GND | Carrier feeds its 3.3 V rail (buck output) in here. This is how the WeAct MCU is powered — 3V3 appears on no other header. |
| P4 | 4 pins | 5V, GND | Unused in Phase 1 (no 5 V rail on the carrier). Leave unconnected; optionally fit a socket and wire only its GND pins for extra mechanical support — do not connect 5V. |
| P1.1 | — | VB (VBAT) | RTC backup supply. Not used in Phase 1 — leave unconnected. |
| P1.2 / P2.2 | — | GND | Signal-header grounds; tie to carrier GND together with the P3 (and any P4) GND pins. |

Power the WeAct from one source at a time — the carrier's 3.3 V through P3, **or** USB — never both at once. The WeAct's on-board LDO (ME6231A33) can back-feed if both are live. A deployed node has no USB, so this only concerns bench work.

### Carrier ↔ WeAct connector interface

The carrier mates with the WeAct board through three sockets, all 2.54 mm pitch:

| Socket | Mates WeAct | Carries |
|---|---|---|
| 2×12 | P1 | signals, VB, GND |
| 2×13 | P2 | signals, GND |
| 2×3 | P3 | 3.3 V power feed, GND |

P1 and P2 are different lengths — P2 has one extra rung (13 vs 12). Column spacing P1↔P2 is 25.40 mm per the WeAct board outline drawing; take exact placement of all three sockets from the WeAct footprint. Use through-hole sockets so the headers bear no mechanical load, and add corner standoffs. A socket on P4 is optional and mechanical-only.

## Free / spare pins

PB3, PB4, PB5, PC0, PC1, PC2, PC3 — 7 pins unassigned.

- PC0–PC3 are ADC-capable; keep as reserve for additional analog channels.
- PB3 is also JTDO / SWO; leave it free if SWO trace debugging may be wanted.

## Sensor-module headers (2×12 + 2×8) — pinout

Every sensor module (M01–M05) mates with the carrier through **two** 2.54 mm headers — the same two connectors and the same pinout across all module types (ADR-0014 decision 5). This is the carrier ↔ sensor-module interface, separate from the WeAct sockets.

The two headers are deliberately **different sizes — 2×12 and 2×8**, so a module seats one way only: the size asymmetry is the mechanical key (a single 2×20 was keyed only by its pin-1 marker). Placed near the two ends of the board, the pair also supports the module at two points — light modules then need no standoffs. Keep mounting-hole provision for the heavier modules (M03, M04) and populate standoffs only where required.

The split doubles as an EMC boundary: the analog ADC inputs are on Header B, on a different connector from the PWM and SPI switching on Header A.

The **signal set** is fixed by ADR-0014. The **pin positions** below are a layout proposal: power at the pin-1 end, ground returns interspersed, each ADC pin flanked by GND. The binding contract is the signal set plus the rule that the carrier and every one of the five module designs use this identical two-header layout — if you re-order, re-order every board together.

Numbering on each header: pin 1 marked; pins 1/2 form the first rung, odd pins on one row, even on the other.

Layout notes:

- **Different sizes are the key.** 2×12 and 2×8 cannot be interchanged or rotated — the module seats one way only. Do not "even them up" to 2×10 + 2×10; that reintroduces the 180° mis-mate the asymmetry exists to prevent.
- **module-ID straps (Header B pins 3–5)** carry no signal from the carrier — each sensor module ties STRAP_0..2 to 3V3/GND in its own ID pattern (ADR-0014 decision 6); 3V3 (B.1) and GND (B.2) sit next to them for that tie. Carrier side they are inputs on PA5/PA6/PA7.
- **ADC pins (B.10, B.12)** are analog, each flanked by GND, on a different connector from every switching signal. Keep their traces short and clear of noise on both carrier and module.
- **5V (A.2)** is live only if the carrier's 5 V rail is fitted (see power section); otherwise leave it unpopulated on the module side.
- **RESERVED (B.7, B.14)** are unconnected — spare positions for future signals without changing the connectors.
- **Power on both headers** — each carries a 3V3/GND pair, so the module has power at both ends and need not route a long supply trace across itself.
- The pin-by-pin assignment is a proposal; the binding contract is the signal set (ADR-0014) and the identical two-header layout on the carrier and all five module designs. If you re-order, re-order every board together.

## Notes

1. **PWM frequency.** All four PWM channels are TIM3_CH1–CH4, so they share one timer and therefore one PWM frequency, with independent duty cycles. If independent frequencies are required, PC6/PC7 can be reassigned to TIM8_CH1/CH2 — same pins, different timer, firmware change only.
2. **PA15 / JTDI.** PA15 defaults to JTDI at reset. With SWD-only debugging (no full JTAG, as used here), PA15 is free for use as GPIO; JTAG is not used.
3. **WeAct pin column.** Each signal's WeAct-pin entry is its position on the core-board headers — P1 (2×12, pins 1–24) or P2 (2×13, pins 1–26). Positions are from the WeAct V1.1 schematic, cross-checked against the board silkscreen. The carrier ↔ WeAct connector interface and the P3 power feed are described in the power section above.
4. **CubeMX caveat.** CubeMX validates AF mappings against the bare STM32F405 only. It does not know which board the chip is on, so it will not flag the WeAct-occupied pins above (notably PA11/PA12 = USB). Any CubeMX pinout must be cross-checked against the occupied-pins table.
5. **CAN1 placement.** CAN1 is assigned to PB8/PB9. PA11/PA12 are equally valid CAN1 pins at the chip level, but on the WeAct board they are the USB data lines; PB8/PB9 keeps the WeAct USB (DFU) available and avoids a shared net with the USB connector.
